你如何使用 Claude Code

你的主线工作是设计到代码的产线工作流，其中 Figma 多设备适配（Fold/Pad，横竖屏变体）是你最具一致性的重复任务，出现在 11+ 会话里。你的交互风格流程驱动、围绕 skill 展开：与其每次都临时描述任务，不如投入精力构建可复用的 SKILL.md 工作流，再不断审计、压缩和打磨它们。每次成功执行后，你常常做事后回顾——让 Claude 分析为什么某处出了问题（比如键盘删除事件、Menu 组件遗漏、删除对话框错位），并提出最小改动以防再犯。这说明你把每次会话既当作交付，也当作工具的训练信号。

你放手让 Claude 跑长链自动化（TaskUpdate 调用了 308 次、Agent 44 次），但在终点是个眼神犀利的验收者——能抓出诸如错误的 NavigationBar 变体、对话框被居中而非贴底 12dp、自动校验漏掉的组件状态。当 Claude 通过 AskUserQuestion 反复发问时，你直接打断，而不是配合它，传递的信号是你更喜欢它带着合理默认值继续，而不是来要授权。值得一提的是，当 Claude 的诊断过于复杂或偏题时，你会直接顶回去——比如「但是我的多设备适配跑得是对的」逼它重新审视真实执行链，或要求它简化根因分析。

你的沟通简短且双语（中文 commit message、中文重写 CLAUDE.md），并且善于批量打包请求——一个会话可能同时包含 git 冲突解决、Figma 适配、插件解释和 SKILL.md 优化。你能容忍摩擦（字体预加载错误、权限拦截、page context 问题），只要 Claude 自我纠错就行；但你不会容忍不结合上下文的死板规则套用，反复指出 Claude 套用表面规则、却不去查阅设备专属映射的情况。

摩擦与问题

你的 Figma 多设备适配工作中反复出现「规则套用错误」与「执行期失败」，而权限和交互式 prompt 问题则频繁打断你的 CLI 流程。

死板套用规则、不查设备专属上下文
在多设备适配中执行 SKILL.md 规则时，Claude 经常套用表面规则（比如「移除移动端语义」），却不去查设备专属映射文件，导致你不得不反复回顾并修补 SKILL.md。把「变换前先查映射」这一类护栏内置到工作流里，能消除这些反复出现的纠正。
  • Claude 在适配时机械执行「移除移动端语义」规则、删除了键盘，但没意识到键盘其实是「搜索激活态」必要组成部分；之后再加回时尺寸还是错的。
  • Claude 在 Fold/Pad 上把删除对话框竖向居中，而不是按规格贴底 12dp，遗漏了设备尺寸规格，需要事后回顾并更新 SKILL.md。
Figma 执行期错误，需要重试
你的 Figma 适配执行经常踩到字体预加载错误、page context 问题，以及只有在校验阶段才浮现的尺寸/坐标假设错误。在 skill 里加一份更严的预执行清单（字体加载、切页、相对坐标），可以减少很多中途返工。
  • 多次适配会话里出现字体预加载错误，导致文本/变体操作被迫重试。
  • Fold/Pad 竖屏 frame 的 StatusBar 宽度最初是错的；frame 坐标用了绝对而非相对 section 的定位——两类问题都是在校验阶段才被抓到。
CLI 权限与交互式 prompt 卡顿
你的 Bash、Skill 和安装命令反复卡在交互式 prompt 或权限拒绝上，打断节奏、需要手动介入。给包管理器和 CLI 默认带上非交互参数（--yes、--global），并预先放行常用工具权限，可以让这些步骤更顺畅。
  • `npm create vite -y` 和一次全局 skill 安装都卡在了 Claude 无法应对的交互式选择上，需要追加 --yes 标志。
  • 评估和 Figma 工作中，Bash 与 Skill 工具调用因为权限模式被拦截，导致你只能手动跑命令，或在任务中途改配置。
